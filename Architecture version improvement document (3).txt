# Redrob Intelligent Candidate Discovery & Ranking Challenge - Dataset Documentation


This document provides a comprehensive overview of the dataset and the project requirements for the Redrob India Runs Data and AI Challenge.


## 1. Challenge Overview
The objective is to build an intelligent candidate ranking system for Redrob AI. You are tasked with analyzing a pool of 100,000 candidates and identifying the **top 100 best-fit candidates** for a specific "Senior AI Engineer" job description. The final output must be a ranked CSV file with scores and reasoning for each of the top 100 picks.


## 2. Dataset Files Provided
The challenge bundle contains the following key files:
- **`candidates.jsonl.gz`**: The primary dataset containing 100,000 candidate profiles in JSONL format (~465 MB uncompressed).
- **`sample_candidates.json`**: A small sample of 50 candidates for quick schema inspection.
- **`candidate_schema.json`**: The JSON schema defining the structure of candidate profiles.
- **`job_description.docx`**: The target JD you are matching candidates against.
- **`redrob_signals_doc.docx`**: Explanation of 23 behavioral signals provided for each candidate.
- **`submission_spec.docx`**: Rules, evaluation criteria, and formatting rules.
- **`sample_submission.csv`**: An example formatting reference for the final output.
- **`validate_submission.py`**: A python script to locally validate your final CSV against format requirements.
- **`submission_metadata_template.yaml`**: Template for metadata to include in your final repository.


---


## 3. Candidate Data Schema & Features
Each candidate is identified by a unique `candidate_id` (format: `CAND_XXXXXXX`) and contains several distinct feature categories:


### A. Core Profile (`profile`)
Basic professional information including:
- **Headline & Summary**: Text descriptions of the candidate's professional identity.
- **Location & Country**: Geographical data.
- **Experience**: Total `years_of_experience`.
- **Current Status**: `current_title`, `current_company`, `current_company_size`, and `current_industry`.


### B. Career & Education History (`career_history`, `education`)
- **Career History**: Array of up to 10 past roles containing `company`, `title`, `start_date`, `end_date`, `duration_months`, `industry`, `company_size`, and a text `description` of responsibilities/achievements.
- **Education**: Up to 5 degrees containing `institution`, `degree`, `field_of_study`, `start_year`, `end_year`, `grade`, and an institution `tier` (tier_1 to tier_4).


### C. Skills & Extras (`skills`, `certifications`, `languages`)
- **Skills**: Array detailing skill `name`, `proficiency` (beginner to expert), `endorsements`, and `duration_months` (how long the candidate has used it).
- **Certifications & Languages**: Assorted arrays detailing formal certifications and language proficiencies.


### D. Behavioral Signals (`redrob_signals`)
A crucial differentiator. These 23 signals simulate real-world platform engagement, acting as modifiers for a candidate's actual availability and quality:
1. **Engagement**: `last_active_date`, `recruiter_response_rate`, `avg_response_time_hours`.
2. **Platform Activity**: `profile_completeness_score`, `applications_submitted_30d`, `interview_completion_rate`, `offer_acceptance_rate`.
3. **Visibility**: `profile_views_received_30d`, `search_appearance_30d`, `saved_by_recruiters_30d`.
4. **Logistics**: `notice_period_days`, `expected_salary_range_inr_lpa`, `preferred_work_mode`, `willing_to_relocate`.
5. **Technical Signals**: `skill_assessment_scores` (Redrob platform scores 0-100), `github_activity_score` (-1 to 100 based on commits/PRs/stars).


> [!WARNING]
> **Behavioral Modifiers**: A candidate with a perfect technical fit but a 6-month-old `last_active_date` or 5% `recruiter_response_rate` is effectively unavailable. Your ranking model **must** down-weight or penalize candidates with poor behavioral signals.


---


## 4. The Target Job Description
You are ranking candidates for the **Senior AI Engineer — Founding Team** role.
**Key Requirements:**
- **Core Experience**: 5–9 years total, with 4-5 years in applied ML/AI at product companies (not pure services/consulting).
- **Hard Skills**: Production experience with embedding-based retrieval (e.g., OpenAI embeddings, sentence-transformers) and vector databases (Pinecone, Qdrant, Milvus, etc.).
- **Evaluation Expertise**: Hands-on experience designing frameworks for ranking systems (NDCG, MAP, offline-to-online correlation).
- **Vibe/Cultural Fit**: "Scrappy product-engineering attitude" over pure research. Needs to ship code fast rather than endlessly theorize.


> [!TIP]
> **Semantic Search, Not Keyword Matching**: Do not simply search for AI keywords in the skills section. A candidate might not have the word "RAG" in their profile, but their career history might show they built a recommendation system at a product company (which is a strong fit). Conversely, someone with all AI keywords but a "Marketing Manager" title is a poor fit.


---


## 5. Project Requirements & Constraints
To build a valid submission, your system must adhere to strict computational and structural rules.


### Compute Constraints
Your final ranking inference pipeline will be reproduced by judges and must fit within:
- **Total runtime**: ≤ 5 minutes wall-clock.
- **Memory**: ≤ 16 GB RAM.
- **Compute**: CPU only (No GPUs allowed during ranking).
- **Network**: **OFF**. No external API calls (e.g., OpenAI, Claude) during the ranking step.
- **Disk**: ≤ 5 GB intermediate state.


> [!IMPORTANT]
> **Pre-computation is allowed**: You can use LLMs, GPUs, or APIs to generate embeddings, indexes, or smaller fine-tuned models *beforehand*, provided that your final ranking script runs fully locally and offline within the 5-minute/16GB CPU limits.


### Artifacts Required for Submission
1. **The CSV Submission**: `team_xxx.csv` containing columns `candidate_id`, `rank`, `score`, `reasoning`.
2. **Code Repository**: A GitHub repo with your source code, setup instructions, pre-computed artifacts (or a script to get them), `requirements.txt`, and a single script command to generate the submission CSV.
3. **Sandbox Demo**: A hosted environment (HuggingFace, Streamlit, Colab, etc.) where your ranker can be run on a small candidate sample to verify reproducibility.


---


## 6. Evaluation Metrics & Pipeline


### Quantitative Scoring (Hidden Ground Truth)
The final composite score is calculated as:
- **0.50 × NDCG@10**: Quality of your top 10 picks.
- **0.30 × NDCG@50**: Quality of your top 50 picks.
- **0.15 × MAP**: Precision across all ranks.
- **0.05 × P@10**: Fraction of top 10 that are relevant.


### The Evaluation Stages
1. **Format Validation**: Automated checks for 100 unique rows, proper headers, decreasing scores.
2. **Scoring**: Against the hidden ground truth.
3. **Code Reproduction & Honeypot Check**: Top-N are run in the sandboxed environment to ensure adherence to compute constraints. Disqualification if the honeypot rate is >10% in the top 100.
4. **Manual Review**: Judges evaluate the `reasoning` column for hallucinations, templated answers, and actual connection to JD requirements.
5. **Defend-Your-Work Interview**: Top X finalists do a 30-minute code walk-through to prove they built the system.


---


## 7. Crucial Pitfalls & Traps (Honeypots)
> [!CAUTION]
> **Honeypot Candidates**: The dataset includes ~80 "honeypot" candidates with impossible profiles (e.g., 8 years of experience at a company that is only 3 years old, or "expert" proficiency in 10 skills with 0 years used). If your system relies solely on keyword embeddings and fails to parse reality, it will rank these honeypots highly. **If your top 100 includes >10% honeypots, you will be disqualified at Stage 3.**


### Reasoning Column Best Practices
The `reasoning` column in your CSV is technically optional but **highly recommended**. When creating it:
- Mention specific facts (years of experience, locations).
- Connect to JD requirements ("strong product ML background").
- Acknowledge gaps (e.g., "high notice period of 90 days, but great technical fit").
- **Never hallucinate skills** not present in the candidate's profile.
- Vary the reasoning text (do not use generic templated strings).

1.
	The 3-Layer "Recruiter-Agent" Architecture
Layer 1: The JD Deconstructor (LLM Information Extraction)
You never feed a raw job description into a search engine. They are filled with corporate garbage ("We are a fast-paced family...").
1. Pass the raw JD into a fast inference API (Groq Llama 3).
2. Instruct the LLM to extract a strict JSON schema:
   * non_negotiable_skills: (e.g., Python, Docker)
   * soft_behavioral_signals: (e.g., led a team, worked in high-ambiguity environments)
   * experience_weight: (Does this role need a senior leader, or a hungry junior?)
Layer 2: Hybrid Retrieval Engine (The Wide Net)
You need to filter down the hundreds of candidates to the top 30 before you do heavy reasoning. You will use a Hybrid Search approach.
1. Dense Vector Search: Embed the candidate's history and match it against the JD's semantic meaning using a local embedding model or OpenAI's API. Store this in Qdrant, Pinecone, or simply a local FAISS index (since it's a hackathon).
2. Sparse Search (BM25): You still need exact keyword matching for hard requirements. If the JD requires "Kubernetes", semantic search might pull someone who knows "Docker." BM25 ensures the hard technical constraints are respected.
3. Combine these scores mathematically: Final_Search_Score = (Vector_Score * 0.7) + (BM25_Score * 0.3).
Layer 3: LLM-as-a-Judge (The "Great Recruiter" Layer)
This is where you win the hackathon. Take the top 30 candidates from Layer 2 and feed their full JSON profile + the Deconstructed JD into your LLM via an engineered prompt. You instruct the LLM to act as a Senior Technical Recruiter and output a final JSON object containing:
1. Culture_Fit_Score (1-10)
2. Technical_Trajectory_Score (1-10) - Is this person getting promoted quickly, or stagnating?
3. Recruiter_Justification: A brutal, 3-sentence summary of exactly why this person is a fit and what their biggest red flag is.
The Tech Stack
Stick to what you can build rapidly and deploy without crashing.
* Orchestration: Python, FastAPI.
* Inference: Groq API (for the LLM-as-a-Judge layer. It's lightning-fast and perfect for parsing 30 resumes in seconds).
* Search / Embeddings: Qdrant (you can run it locally in Docker) or a local FAISS index using SentenceTransformers.

If I had to prioritize effort:

Problems we could have to keep in mind: 




1
	Your architecture is now heavily dependent on LLMs in a few critical places. If you don't understand where probabilistic behavior can hurt you, you can spend days tuning a model only to discover your labels are unstable.
Let's audit the entire pipeline.
________________


Rule of Thumb
The biggest danger is:
If the same candidate gets different scores when the prompt is run twice, your entire ranking pipeline becomes noisy.
For example:
Run 1:
{
 "candidate": "A",
 "score": 92
}
Run 2:
{
 "candidate": "A",
 "score": 78
}
Now your LightGBM is learning from inconsistent supervision.
________________


Phase 1 — JD Intelligence Engine
Risk Level: LOW
You only run this once.
The JD is fixed.
Even if:
Run 1:
ownership_score = important


Run 2:
leadership_score = important
the impact is small.
________________


Potential Errors
Requirement Extraction Drift
Example:
JD:
Experience with ranking evaluation.
LLM extracts:
{
 "evaluation": false
}
or misses NDCG/MAP entirely.
________________


Fix
Run:
3-5 independent parses
and merge results.
Treat this like ensemble extraction.
________________

The Biggest Insight
For this competition, the "hyperparameter problem" is not:
learning_rate = 0.03 vs 0.05
The real hyperparameter problem is:
How much retrieval?
How much teacher influence?
How much behavioral influence?
How much anomaly penalty?
Those decisions will move your leaderboard far more than classic model tuning. That's why your tuning system should be built around retrieval quality, teacher quality, label quality, and ranking stability, with model hyperparameters treated as the final optimization step rather than the starting point.
