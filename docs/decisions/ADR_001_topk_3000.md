# ADR 001: Top-K Retrieval Set to 3000

## Decision
We have hardcoded `retrieval.top_k = 3000` as the fixed boundary between the Retrieval subsystem and the Ranking subsystem.

## Context
Standard RAG systems retrieve Top-100. However, our downstream ranking ensemble (LightGBM/XGBoost) requires sufficient negative and edge-case samples to learn the complex, non-linear boundaries of a "founding engineer". 

## Alternatives Considered
- **Top 1000**: Too few samples. The LTR model overfits on the elite candidates and fails to learn what a "good but not great" candidate looks like.
- **Top 5000+**: Computationally prohibitive. Phase 7 (Teacher Ensemble) requires calling the LLM 3 times per candidate. 5000 candidates = 15,000 LLM calls per training run.

## Consequences
- **Positive**: 3000 guarantees excellent recall while keeping LLM budget and pairwise generation costs within the Kaggle competition constraints.
- **Constraint**: This value is LOCKED. Changing it invalidates all calibrated labels and requires a full pipeline retrain.
