# Phase 2 Implementation Plan: Domain Ontology Engine

## Objective
Encode expert recruiter knowledge about the "Senior AI Engineer — Founding Team" role into frozen, deterministic keyword dictionaries. This acts as the backbone for the sparse retrieval and feature engineering stages downstream.

As per the finalized V12 architecture rules, **this is a manual artifact and will NOT be generated by an LLM at runtime.**

## Proposed Implementation

### 1. File Structure
Create `offline/phase02_ontology_engine.py`

### 2. The Frozen Ontology
We will create a hardcoded `ONTOLOGY` dictionary. This dictionary maps core JD concepts to their synonym and keyword lists.

**Core Ontology Groups:**
- `RETRIEVAL`
- `VECTOR_DB`
- `EMBEDDINGS`
- `EVALUATION`
- `PRODUCT_COMPANY`
- `FOUNDING_TEAM`
- `OWNERSHIP`
- `ML_AI`

**Anti-Bias Rule (Zero Generic Company Names):**
The ontology mapping for consulting vs product background will strictly evaluate the *nature* of the work (e.g., "IT services", "outsourcing") and completely avoid generic company names (like "TCS", "Infosys") to prevent biased penalization of engineers.

### 3. Core Logic
1. The script will convert all ontology terms to lowercase for fast, case-insensitive substring matching.
2. It will save the resulting mapping to `data/artifacts/ontology.json`.
3. It will expose an exact function `compute_ontology_scores(text: str) -> dict` which calculates the coverage (percentage of terms matched, scaled).

```python
def compute_ontology_scores(text: str) -> dict:
    """
    Returns a dict of group_name -> coverage_score (0.0 to 1.0)
    for each ontology group based on how many terms appear in text.
    """
    text_lower = text.lower()
    scores = {}
    for group, terms in ONTOLOGY_LOWER.items():
        matches = sum(1 for term in terms if term in text_lower)
        scores[group] = min(1.0, matches / max(1, len(terms) * 0.3))
    return scores
```
